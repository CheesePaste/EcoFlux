Ecoflux 代码审查锐评——用大白话吐槽这个项目到底写了些什么玩意

前排提示：以下内容纯属技术吐槽，不是人身攻击，代码写烂了改就完了，别破防。


============ P0 级别：这代码真的能跑？ ============


1. 骨架每帧都在重生，CPU 在哭泣

TreeGrowthSession.java 第 55 行：

    if (skeleton != null && morphologyParams == params) return;

你看看这个 ==，你再看看 MorphologyParams.oak() 每次返回 new 新对象。
这行代码的意图是"如果参数没变就别重新生成骨架了"，实际效果是"每次必定重新生成"。
相当于你设了个闹钟说"如果天没亮就别叫我"，然后你住在北极。
树每长一个阶段就从头跑一遍递归分枝算法，CPU 风扇就是被你这样干废的。

锐评：用 == 比较两个对象是不是同一个，这是 Java 入门的第二天就会讲的事。
你猜怎么着，写这个的人可能第一天翘课了。


2. 树的高度到底是多少？六棵树给出了十二个答案

来看看橡树：profile 说我 5 格高，MorphologyParams 说我 6 格高。
白桦：profile 说我 6，params 说我 7。
云杉：profile 说我 8，params 说我 10。
金合欢：profile 说我 5，params 说我 6。

到底谁说了算？嗯，取决于你是从哪个路径调进来的。
resolveHeight() 听 profile 的，totalStagesForHeight() 听 params 的。
相当于你问一个人他多高，他说"看情况，我站直了 170，弯着腰 165"。

锐评：同一个数据在两个地方各存一份还不一样，这不是 bug 什么是 bug。
建议给每个树配一个身高造假证明。


3. 服务端觉得这是一朵花，客户端觉得它什么都不是

服务端 SimplePlantAdapter：我认识高草丛、大型蕨、高花、蘑菇，我们都是好朋友。
客户端适配器：什么高草丛？什么蘑菇？我只有矮花和矮草和树苗，其他的交给通用兜底！

于是 TALL_GRASS 被服务器当正经植物追踪，到了客户端喜提 GenericVisualLifecycleAdapter，
获得一份"万能通用视觉配置"，跟你去裁缝店说"随便改改"然后拿到一条麻袋差不多。
而且死灌木在客户端被当作草渲染，在服务端根本不认识它——这是一个单向暗恋。

锐评：服务端和客户端的植物匹配逻辑是两个人分别写的，写完了之后互相没看过对方的代码。
建议两个人打一架，赢的人决定哪些植物应该被追踪。


4. 树长完了，回头找自己，发现自己已经死了

TreeGrowthHandler：树长完了！让我看看这是什么树……诶我刚刚不是把你从 map 里删了吗？
于是 get(basePos) 返回 null。
于是 null check 触发。
于是回退到 OAK_LOG。
不管你是云杉还是丛林还是金合欢，长完的那一刻，你都是橡木。

锐评：先把自己删了，然后再查找自己，发现找不到，然后说"那就默认吧"。
这个逻辑跟出门把钥匙锁屋里然后打 110 说自己是小偷有什么区别。


============ P1 级别：复制粘贴工程师的人才作品 ============


5. 同样 9 行代码，Ctrl+C Ctrl+V 了三遍

BiomeTransitionService.applyTransition() 里面有一段 9 行的数据重置代码。
applyRegression() 里面——一模一样，一个字都不差。
SuccessionTargetResolver.resolveTarget() 里面——又来了，熟悉的 9 行。

这叫什么？这叫"我不信任你写的 clearRuntimeState() 方法，所以我自己再写一遍"。
而 clearRuntimeState() 确实不值得信任——因为它只清了一半的字段。
剩下一半留给调用者自己动手，丰衣足食。

锐评：如果你以后要加一个需要重置的新字段，你得改四个地方。
漏一个就有 bug。猜猜谁在凌晨三点改完三个地方觉得自己完事了。


6. 生命周期时长？写了三遍，以防万一

同样一个"植物出生阶段 200 tick"这件事：
PrototypeChunkController 里写了一遍。
SimplePlantAdapter 里写了一遍。
SaplingAdapter 里又写了一遍（还是不一样的值，因为树苗有自己的想法）。

三个地方独立维护，改一个不改另外两个，普通模式和加速模式的行为就分道扬镳。
这相当于三个不同部门各维护一份公司制度，然后发现大家的午休时间都不一样。

锐评：常量化了解一下？共享常量类了解一下？算了，Ctrl+C 已经按出肌肉记忆了。


7. 三百七十行死代码，致敬曾经的自己

所有 6 个 tree profile（橡树到金合欢）都有两个 growStage()——新版用 Morphology 系统的，旧版用直杆+圆盘的。
新版写好以后，旧版从来没有被调用过。
TreeGrowthHandler 里有一个 if (morphologyParams != null) { 新路径 } else { 旧路径 } 的分支。
所有 6 棵树 morphologyParams 都不为 null，旧路径是纯纯的死代码。
还有 TreeShapeUtils 里的 oakCanopyRadius、birchCanopyRadius、spruceCanopyRadius、generateBranchPositions，
全是被旧代码养着的，旧代码死了它们就是代码界的陪葬品。

锐评：写新功能的时候舍不得删旧代码，说"万一还要用呢"。
兄弟，git 有历史记录，你不是在写 Word 文档不用另存为。


8. GrowthBlockCapture——一个从来没人调用的捕鼠器

GrowthBlockCapture 这个类设计了一个精妙的 ThreadLocal 机制来捕获方块放置。
LevelMixin 里专门写了个 Mixin 去拦截所有 Level.setBlock() 调用。
然后在全代码库里搜索谁给 ACTIVE 设为了 true……
零。
这个捕鼠器摆在厨房三个月了，从来没触发过，老鼠在它旁边开派对。

锐评：写了一段代码、写了一个 Mixin、甚至写了一个类，然后忘了写"启动"按钮。
这不是未完成的功能，这是行为艺术。


9. ModPlayerEvents 说：VegetationTracker 你写的我不放心，我自己来

VegetationTracker 有个 trackAt() 方法，做了 1. 找 adapter 2. captureBirth 3. 存入 vegetationRecords 4. 网络同步。
ModPlayerEvents.onBlockPlaced() 把这四个步骤原样重新实现了一遍。
VegetationTracker 有个 untrack() 方法，做了从 vegetationRecords 删除 + 同步。
ModPlayerEvents.onBlockBroken() 又把这原样实现了一遍。

这相当于你妈给你做了饭放在桌上，你说"谢谢"，然后自己下了一碗一模一样的面。

锐评：如果将来 trackAt() 要加一步"记录日志"或者"触发事件"，
ModPlayerEvents 会安静地崩掉因为没人记得去改那个复制品。


10. SuccessionService 的两个方法，其实是一个

step() 和 processChunkTick() 的核心逻辑——prune → queue → spawn → observe → evaluate → transition——
是一字不差的同一套流水线。区别只是 step() 说 ignoreInterval=true，processChunkTick() 说 false。

相当于你写了一个"吃面"函数和一个"用筷子吃面"函数，内部代码一模一样，
只是传了一个参数叫 useChopsticks。

锐评：这两个方法就差一个 boolean 参数。考虑到还要留两个 public 方法当接口，
你只需要一个 private 方法干实际的事，两个 public 当皮包着就行了。
现在这样属于是代码肥胖症。


11. 植物有七个生命阶段，但客户端说：我只认识四个

服务端：BORN → JUVENILE → GROWING → MATURE → AGING → DEAD → TRANSFORMED（七个阶段，讲究）
客户端：BORN → GROWING → MATURE → AGING（四个阶段，下班）

映射逻辑：
JUVENILE？那就算 BORN 吧，刚出生和青春期能有什么区别。
TRANSFORMED？那就算 MATURE 吧，刚变身的树和老树能有什么区别。
DEAD？那就算 AGING 吧，死了和老了能有什么区别。

锐评：你搁这做阶段合并呢？语义精度损失在每一步都是理直气壮的。
等哪天策划说"我想让死掉的树看起来不一样"，你就得把这套映射全推翻重来。


12. 客户端视觉时长数值，全靠心算对齐

服务端说植物 BORN 阶段 200 tick。
客户端也在自己的 VisualLifecycleProfile 里写了 born=200。
服务端说 GROWING 阶段到 1200 tick。
客户端写了 growing=1000（意味着 born+growing = 1200，还得靠心算）。
服务端没有给客户端发时长配置，客户端也没从服务端要。
全靠两边的程序员对拍，口头约定"咱们阶段时长保持一致啊"。

锐评：这叫"约定优于配置"的反面——"约定全靠运气"。
哪天有人改了服务端的时长忘了通知客户端，两边就静默不一致，而且只有肉眼能看出来不对劲。


============ P2 级别：设计品味有待提升 ============


13. 默认颜色逻辑：复制一遍才放心

VisualLifecycleClientRuntime 和 ModClientVisualLifecycle 各有一份"这个方块应该是什么颜色"的逻辑。
两份代码的功能完全一样，魔法颜色 0xA78F63 在两处各写了一次。
你觉得这个颜色代表什么？死灌木的棕色。你能从 0xA78F63 看出来吗？看不出来就对了。

锐评：提取常量是你跟代码之间的基本尊重。现在这颜色值直接糊在脸上，属于代码界的"猜猜我是谁"。


14. 群系转换种的树，全是橡木和白桦

成功演替后要种几棵树当奖励。种什么树？代码告诉你了：
    randomTreeBlocks() = [OAK_SAPLING, BIRCH_SAPLING]
从平原变森林？种橡木和白桦。从森林变黑森林？种橡木和白桦。从沼泽变红树林？种橡木和白桦。

锐评：不管演替目标是什么群系，奖励永远只有 Oak 和 Birch 两个选择。
在配置文件里辛辛苦苦设计了 21 条演替路径，最后种树这里直接开摆。
建议把 BiomeTransitionService 改名叫 "OakAndBirchEverywhereService"。


15. progress() 函数被复制粘贴了三次

SimplePlantAdapter、SaplingAdapter、TreeStructureAdapter 里面都有一个完全相同的：
    private static float progress(long age, long start, long endExclusive)
三个文件，三份拷贝，逐字不差。

锐评：Java 8 的接口 default 方法都出来十年了。
你哪怕写个工具类呢？写个 VegetationUtils 呢？
非得 Ctrl+C Ctrl+V，下次改算法的时候漏改一个，又要排查一整天。


16. JSON 配置里三个字段是纯纯的占位符

evaluation_interval_days.max — 解析了，存了，评估器从来没看过。
spawn_rules.placement — 解析了，存了，findSpawnPos 从来没理过。
plants[].category — 解析了，存了，全游戏逻辑里没有一处读取。

每一个 JSON 文件都老老实实填了这些字段，21 个文件，每一行都是虔诚地给虚空打工。

锐评：你是不是在凑 JSON schema 的字数？哥，没人评审你的 JSON，不用的字段删了吧。


17. ActiveVegetationRecord.category — 存了个寂寞

这个 category 字段被写入 NBT，被读回来，被调试命令打印，
但从来没有哪个 if 语句因为它走不同的分支。点值计算不看它，评估不看它，修剪不看它。
它存在于整个系统中唯一的理由是"多一个字段看起来更专业"。

锐评：数据库里有一列从来不查，这就是典型的"万一用得上"后遗症。
万一用不上呢？万一你只是在浪费 NBT 空间呢？


18. 蘑菇兄弟俩：代码相似度 80%

BrownMushroomGrowthProfile 和 RedMushroomGrowthProfile。
placeStem() — 完全一样（都往蘑菇柄上放 MUSHROOM_STEM）。
totalStagesForHeight() — 完全一样。
canGrowStage() — 几乎一样（把 BROWN 换成 RED）。
整个文件就差把 "Brown" 全局替换成 "Red" 了——啊不对，有人已经这么干了。

锐评：多态和继承这两个概念是 Java 的核心卖点。
你猜怎么着，写蘑菇的人直接从布朗蘑菇文件另存为红色蘑菇文件，改都不用改多少。
抽象基类：我不配吗？


19. canGrowStage() 五个 profile 里写的一模一样

Oak/Birch/Spruce 的 canGrowStage() 都是检查单格位置能不能长。
Jungle/DarkOak 的另一个版本，但彼此相同。
TreeGrowthProfile 是一个接口。
接口可以有 default 方法。

锐评：Java 8 发布那年是 2014 年。现在 2026 年了各位。


20. ModCommands 290 行，switch 套了 12 个 case

一个类、290 行、一个 switch、12 个分支。
每个分支里混着逻辑代码和格式化输出。
PLANTS case 重新实现了 PlantSpawner 已经有的一段逻辑。
因为"反正就几行，复制过来更快"。

锐评：这个类迟早涨到 500 行。到时候每一个 case 都是一个雷。
建议现在拆，别等它变成 legend 再动手。


============ P3 级别：小毛病但很扎眼 ============


21. SuccessionConfigRegistry 的 volatile 套娃 synchronized

三个字段标了 volatile，方法标了 synchronized。
synchronized 自己就带 memory barrier，volatile 是多余的。
这就像穿了秋裤又套了棉裤——虽然不犯法，但走在街上会被人看。

锐评：一看就是"并发编程面试题背多了，不确定用哪个，那就都用上"。


22. 死常量：ACTIVE_PLANTS

SuccessionChunkData 里有个 private static final String ACTIVE_PLANTS。
当前 NBT 序列化早就不用这个 key 了。它孤独地挂在类的最上面，像一个没人记得的纪念碑。

锐评：疑似 2025 年留下的文物，考古价值大于实用价值。


23. JungleGrowthProfile import 了两次 java.util.List

第 8 行和第 8 行——哦不，第 8 行到第 10 行之间，java.util.List 出现了两次。
IDE 没报 warning 是因为你关了？还是你根本不在 IDE 里写 Java？

锐评：这种错误小学生改完代码跑 lint 都能发现。你甚至不需要 lint，你有眼睛就行。


24. SkeletonGenerator 有个变量叫 currentPos

第 58 行赋值一次，第 101 行又赋值一次。
在两次赋值之间，没有人读过它。它存在的唯一意义是让读者思考"我是不是漏看了什么"。

锐评：变量界的占着茅坑不拉屎。


25. 文档写了不存在的类

tree-growth-system.md 第 173-199 行深情描述了 BlockDisplayAnimator 如何用 Display 实体加反射做动画。
事实：这个类不存在。实际实现在 ClientGrowthAnimationManager，用的是直接方块渲染。
文档和代码的关系大概相当于产品经理的需求文档和程序员最终发布的功能。

锐评：文档写得太好了，建议把代码按文档重写一遍。


26. speedMultiplier 是全局可变静态变量

    public static int speedMultiplier = 1;

至少 4 个不相关的子系统在读取它。
任何一个地方改了它，另外三个地方的行为全变。
这是什么设计模式？这叫"全局变量飞线模式"。

锐评：如果将来要支持两个不同速率的维度同时运行——算了，这个项目看起来不会走那么远。


============ 总结 ============

这个项目一共有：
- 4 个严重 bug（其中 2 个是运行时正确性错误，1 个性能退化，1 个数据不一致）
- 8 个高危重复（改一个地方必须改 2-5 个其他地方否则出 bug）
- 13 个中等问题（死代码、死字段、设计不一致）
- 7 个小毛病

代码里有多少行？不清楚。
死代码有多少行？大约 450 行（旧 growStage + GrowthBlockCapture + 未使用字段）。
Ctrl+C Ctrl+V 产生的重复有多少处？保守估计 30+ 处。

最经典的操作：
- 用 == 比较对象 → 循环优化直接废了
- 同一个数据在 6 个 profile 里各存了不同版本
- 服务端 7 个生命阶段到客户端变成 4 个，"因为不需要那么细"
- 三个地方写同一段重置代码，一个 clearRuntimeState 只清了半截
- JSON 里填了 3 个不被使用的字段，21 个文件没一个落下

如果这是一份代码作业，评分：功能跑起来了 +10 分，架构设计 -5 分，重复代码 -5 分，文档过时 -3 分，总分不挂科但也没奖学金。

好消息：这些问题没有一个修复起来特别难。大部分是"删掉"、"合并"、"提取常量"级别的操作。
坏消息：不改的话，下一个接手的人会在凌晨三点对着你三年前的复制粘贴骂娘。

锐评完毕，改不改你看着办。
